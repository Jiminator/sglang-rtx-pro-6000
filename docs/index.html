Node topology — g4-standard-384 (dual NUMA)

A g4-standard-384 node is a dual-socket AMD Turin machine. Each NUMA domain owns half the vCPUs and memory, four GPUs on PCIe Gen 5, and one of the two 200 Gbps SmartNICs. There is no NVLink — cross-GPU traffic rides PCIe within a socket and the inter-socket fabric across NUMA domains.

Why the recipes care. Logical topology shown; exact lane mapping can vary by platform. The dual-NUMA, dual-NIC layout explains three recurring flags: SGLANG_SET_CPU_AFFINITY=1 pins each GPU's worker to its local socket, NCCL_SOCKET_IFNAME=eth0,eth1 (and DRANET's eth1/eth2) spreads inter-node traffic across both NICs, and NCCL_P2P_LEVEL=SYS permits peer transfers that cross the socket link since there's no NVLink.

Native 4-bit floating point (NVFP4)

SM120 introduces hardware NVFP4 (E2M1: 1 sign, 2 exponent, 1 mantissa bit) with micro-tensor scaling — two-level scaling using 16-element blocks with FP8 scale factors plus a global FP32 scale — to manage dynamic range. Supported formats span FP4, FP6, FP8, INT8, FP16, BF16, TF32, and FP32. This is why the modelopt_fp4 recipes in these docs can fit trillion-parameter-class MoE models on a single 8-GPU node.

Titanium offload architecture

G4 instances offload infrastructure work to dedicated silicon, keeping host CPUs free for tokenization and scheduling:

  • Networking — Titanium Offload Processors (TOPs) handle VPC networking and packet processing for up to 400 Gbps, with Andromeda virtual-network encapsulation offloaded and line-rate encryption.
  • Storage — Titanium Local SSD up to 12 TiB with a hardware-accelerated data path that bypasses the host CPU.

Default shape. Every multi-GPU recipe in these docs targets g4-standard-384 (8× RTX PRO 6000, 1.5 TB RAM, 400 Gbps). Multi-node setups simply use two of them.

Implications for SGLang

  • 96 GB per GPU allows high-density KV-cache allocation — recipes run --mem-fraction-static at 0.85–0.90.
  • NVFP4 serving rides the 5th-gen tensor cores for 4-bit inference (--quantization modelopt_fp4).
  • No NVLink — GPU-to-GPU traffic for TP/PP goes over PCIe Gen 5, which is why every recipe sets NCCL_P2P_LEVEL=SYS, disables InfiniBand, and disables custom all-reduce in favor of FlashInfer's fused all-reduce.

GKE cluster setup

Provision a GKE environment purpose-built for distributed SGLang: high-MTU VPC, multi-networking, private Blackwell node pools, and an isolated pool for the load generator. One script does all of it.

What the template provisions

  • A custom VPC with MTU 8896 for optimized multi-node traffic
  • Two subnets with secondary ranges for Pods and Services
  • Firewall rules for internal traffic, SSH, and master→node communication
  • A Cloud Router + NAT gateway so private nodes can reach the internet
  • An Artifact Registry repository for Docker images
  • A private GKE cluster with multi-networking enabled
  • A GPU node pool of g4-standard-384 (8× RTX PRO 6000 each, default 2 nodes, scalable to 3)
  • A benchmark-client node pool (n2-standard-64) so load generation never contends with GPU serving

Run the script

  1. Set the required project variable.

    The script refuses to run without it, to prevent accidental creation in the wrong project.

    shell
    export PROJECT_NAME="your-gcp-project-id"
  2. Optionally override defaults.
    shell
    export REGION="us-central1"
export ZONE="us-central1-a"
export CLUSTER_NAME="my-custom-sglang-cluster"
  3. Review CIDR ranges (critical).

    Open the script and check the defaults in section 1 against your existing networks. Defaults: subnet 10.0.0.0/20, pods 10.1.0.0/20, services 10.1.16.0/20, master 172.16.0.16/28.

  4. Execute.

    The script is idempotent — each resource is checked and skipped if it already exists.

    shell
    chmod +x createCluster_template.sh
./createCluster_template.sh

Benchmarking guide

The three-step workflow used to produce every result in these docs: deploy the server, run an isolated load generator, extract the JSON. The standard profile is 1024-in / 8192-out at 512 concurrency.

  1. Deploy the serving framework.

    Single-node models use a Job or single-replica StatefulSet; distributed models use a 2-replica StatefulSet plus headless services for rendezvous.

    shell — examples
    # Single node (DeepSeek NVFP4)
kubectl apply -f models/DeepSeekv3-2/nvp4/sglang-ds32-job-v2.yaml

# Distributed 2-node (GLM-5.1)
kubectl apply -f models/GLM5.1/sglang-glm51-v1-04232026.yaml

    Verify with kubectl logs -f <pod> and wait for The server is fired up and ready!

  2. Run the benchmark client.

    The client pod is pinned to the dedicated CPU pool via nodeSelector: workload: benchmark-client, ensuring zero interference with the GPUs. The canonical client manifest:

    benchmarking_scripts/benchmark-dsv2.yaml
    apiVersion: v1
kind: Pod
metadata:
  name: sglang-benchmark-client
spec:
  nodeSelector:
    workload: benchmark-client   # forces the pod onto the CPU pool
  restartPolicy: Never
  containers:
  - name: benchmark
    image: docker.io/lmsysorg/sglang:v0.5.10.post1
    command: ["/bin/bash","-c"]
    args:
    - |
      API_URL="http://sglang-serving-on-master-opt:8000"
      MODEL_NAME="deepseek-ai/DeepSeek-V3.2"
      CONCURRENCY=512
      INPUT_LEN=1024
      OUTPUT_LEN=8192
      NUM_PROMPTS=1536

      python3 -m sglang.bench_serving \
          --backend sglang-oai \
          --base-url $API_URL \
          --model $MODEL_NAME \
          --dataset-name random \
          --random-input-len $INPUT_LEN \
          --random-output-len $OUTPUT_LEN \
          --num-prompts $NUM_PROMPTS \
          --max-concurrency $CONCURRENCY \
          --output-file /workspace/results.json \
          --apply-chat-template

      echo "Benchmark complete! Sleeping so you can extract results..."
      sleep infinity

    Monitor with kubectl logs -f sglang-benchmark-client.

  3. Extract results.

    The client sleeps after completion so you can copy the output:

    shell
    kubectl cp sglang-benchmark-client:/workspace/results.json ./results.json

Troubleshooting

SymptomFix
Pod fails with OOMLower --mem-fraction-static in the manifest; confirm the model + KV cache fits the 96 GB × GPU budget.
Multi-node hangs at NCCL initThe VPC must use MTU 8896 — use createCluster_template.sh. Check NCCL_SOCKET_IFNAME=eth0,eth1 matches your interfaces.
Benchmark numbers look noisyVerify the client landed on the benchmark pool (kubectl get pod -o wide), not a GPU node.

Metric glossary

MetricMeaning
TTFTTime to first token — dominated by prefill and queueing. At sustained 512-concurrency saturation, median TTFT can legitimately reach minutes for configs without enough prefill headroom.
TPOTTime per output token, excluding the first — the decode-phase pace each request experiences.
ITLInter-token latency distribution across all tokens.
Output / total tok/sAggregate generated (and generated+input) tokens per second across all concurrent requests.
Accept lengthFor speculative decoding: average accepted draft tokens per verification step (higher = more speedup).

Takeaways

  • TTFT tail collapses with HiCache. P99 TTFT drops from 14.05 s to 3.36 s — a 4.2× improvement — and mean TTFT improves ~18%. For interactive 20K-prompt workloads, HiCache is the difference between consistent and occasionally terrible first-token latency.

Agentic benchmark — Kimi-K2.6

Standard random-token benchmarks miss what agents actually do: long multi-turn sequences with heavily shared prefixes and a long tail of steps. This benchmark replays real Kimi-K2.6 agentic traces against SGLang.

The workload

  • Traffic: replay of real-world agentic traces (long-tail step counts and sequences), temperature 0.6, top_p 0.95, max 4096 tokens per step.
  • Why it's hard: long sequences plus a large prompt volume stress KV-cache management; the default benchmark parallelism of 256 overwhelms a single node and causes timeouts, so the single-node run uses parallelism 64 (128 may work but is untested).
  • Data: ships as benchmark_scripts/data.jsonl.zst — decompress with zstd -d before running.

SGLang features that made it work

FeatureRole
HiCache (hierarchical cache)Spills KV cache to host memory (--hicache-ratio 2.0, write_through_selective) so multi-turn prefixes survive eviction — the key to the 81% hit rate.
EAGLE3 speculative decodingDraft model lightseekorg/kimi-k2.5-eagle3 accelerates decode. Dual-node uses a swept setting of (steps 4, drafts 6) vs (3, 4) on single-node.
LPM scheduling--schedule-policy lpm (longest-prefix-match) batches requests to maximize cache reuse.
SMG routerRoutes traffic across the two GCE replicas (power_of_two policy). Planned replacement: GKE Inference Gateway.

    shell

    Enable Workload Identity and the GCS Fuse CSI driver, then grant roles/storage.objectViewer on the weights bucket to both the Kubernetes service account (via Workload Identity) and the default compute service account.

    shell

Network & NCCL benchmarks

G4 nodes carry two 200 Gbps Diorite SmartNICs. The DRANET API plane with an 8896-MTU VPC exposes both to pods natively — and these benchmarks prove you can actually use the aggregate bandwidth.

Headline numbers

DRANET infrastructure

Three pieces: a dedicated 8896-MTU VPC with isolated payload subnets, a Dataplane-V2 GKE cluster whose G4 node pool attaches both extra networks with the DRA networking driver label, and a Kubernetes ResourceClaimTemplate that lets pods claim the NIC devices.

VPC + subnets (01_setup_network.sh essence)
gcloud compute networks create dranet-vpc --subnet-mode=custom --mtu=8896
gcloud compute networks subnets create sub0 --network=dranet-vpc --range=10.10.0.0/16 --region=us-central1
gcloud compute networks subnets create sub1 --network=dranet-vpc --range=10.20.0.0/16 --region=us-central1
gcloud compute networks subnets create sub2 --network=dranet-vpc --range=10.30.0.0/16 --region=us-central1
gcloud compute firewall-rules create allow-dranet-internal \
    --network=dranet-vpc --allow=tcp,udp,icmp --source-ranges=10.0.0.0/8
cluster + DRANET node pool (02_create_cluster.sh essence)
gcloud container clusters create dranet-cluster \
    --network=dranet-vpc --subnetwork=sub0 \
    --enable-ip-alias --enable-dataplane-v2 \
    --zone=us-central1-b --num-nodes=1 --machine-type=e2-standard-4

gcloud container node-pools create g4-dranet-pool \
    --cluster=dranet-cluster --machine-type=g4-standard-384 \
    --accelerator type=nvidia-rtx-pro-6000,count=8 \
    --enable-gvnic --num-nodes=2 --zone=us-central1-b \

Running the benchmarks

  1. Provision. ./01_setup_network.sh && ./02_create_cluster.sh — both idempotent.
  2. Deploy the DRA template. kubectl apply -f 03_deploy_dra.yaml
  3. iperf3: kubectl apply -f 04_iperf_dranet.yaml then ./run_iperf_10x.sh for 10 logged bidirectional runs.
  4. NCCL multi-NIC: apply chiv01-nccl-bench.yaml (the dual-NIC "golden" deployment), then ./run_bench_10x.sh, or run ranks manually:
